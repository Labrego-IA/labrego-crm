## Story

**Como** gestor de vendas,
**Quero** configurar na cadência exatamente o que deve acontecer em cada etapa do funil (ligação, WhatsApp, email), com qual conteúdo e após quantos dias, e que o sistema execute essas ações automaticamente através dos agentes (voice agent, WhatsApp, email),
**Para que** o pipeline de vendas funcione de forma autônoma no caminho de "sem retorno", sem depender de ação manual humana para cada follow-up.

---

## Contexto

### Estado Atual
- `/cadencia` tem **UI completa** para criar steps com branching condicional (~1.513 linhas)
- Steps já armazenam: `contactMethod` (whatsapp/email/phone/meeting), `daysAfterPrevious`, `messageTemplate`, `objective`, branching (`parentStepId` + `condition`)
- Campos no contato já existem: `currentCadenceStepId`, `lastCadenceActionAt`, `lastCadenceStepResponded`
- VAPI calls funcionam via `/api/call-routing/call-contact` com speech customizável
- WhatsApp via Twilio funciona em `callRouting.ts` → `sendWhatsAppMessage()`
- Email via Nodemailer funciona em `email.ts`
- **MAS:** Nenhum scheduler/cron executa steps automaticamente
- **MAS:** `daysAfterPrevious` é armazenado mas **ignorado**
- **MAS:** Response tracking (`lastCadenceStepResponded`) não é atualizado automaticamente
- **MAS:** Branching condicional existe na UI mas **não é aplicado na execução**
- Feature `crm_automation` existe no plano Pro

### Conceito Central — Fluxo Linear de "Sem Retorno"
O usuário definiu a regra de ouro: **quando o contato responde positivamente, avança para a próxima etapa do funil.** Portanto, a cadência trata apenas do **caminho do "sem retorno"** — é um fluxo linear:

```
Contato entra na etapa "Prospecção"
  ↓ Dia 0: WhatsApp de apresentação
  ↓ (sem retorno)
  ↓ Dia 3: Ligação com speech X
  ↓ (sem retorno)
  ↓ Dia 5: Email de follow-up
  ↓ (sem retorno)
  ↓ Dia 10: Última tentativa - WhatsApp
  ↓ (sem retorno)
  → Cadência esgotada (move para etapa de "Sem resposta" ou mantém)
```

Se em qualquer ponto o contato **responde positivamente** → move para próxima etapa do funil → inicia a cadência daquela etapa.

### O que Muda
1. **Redesign da `/cadencia`** — foco em configurar fluxo linear por etapa do funil
2. **Motor de automação (Cadence Engine)** — scheduler que processa steps automaticamente
3. **Conectar os agentes** — VAPI (ligação), Twilio (WhatsApp), Nodemailer (email) executam via engine
4. **Logging completo** — cada ação automática registrada no log do contato
5. **Dashboard de execução** — visibilidade do que está rodando/agendado

## Acceptance Criteria

### Bloco A — Redesign da Página de Cadência

#### AC-1: Visão por funil e etapa
**Given** o usuário acessa `/cadencia`
**When** a página carrega
**Then** vê:
- Seletor de funil no topo
- Lista de etapas do funil selecionado (ordenadas)
- Cada etapa é expandível e mostra seus steps de cadência
- Indicador visual: "X steps configurados" por etapa

#### AC-2: Configuração de steps lineares por etapa
**Given** o usuário expande uma etapa do funil
**When** ele visualiza/edita os steps
**Then** vê um fluxo linear vertical (timeline) com:
- Cada step como um card conectado por linha vertical
- **Timing**: "Após X dias" (editável, default 0 para o primeiro)
- **Canal**: ícone + label (WhatsApp | Ligação | Email | Reunião)
- **Conteúdo**: preview do template/speech
- **Status**: ativo/inativo (toggle)
**And** botão "Adicionar step" no final do fluxo
**And** drag-drop para reordenar steps

#### AC-3: Configuração do conteúdo por canal
**Given** o usuário cria ou edita um step
**When** o modal de configuração abre
**Then** os campos variam por canal:

**WhatsApp:**
- Template de mensagem (texto com variáveis `{{nome}}`, `{{empresa}}`)
- Preview da mensagem com dados de exemplo

**Ligação (Voice Agent):**
- Speech/roteiro do agente (textarea rico)
- Objetivo da ligação (texto)
- Variáveis: `{{nome}}`, `{{empresa}}`, saudação por horário
- Preview: "O agente vai dizer: [texto renderizado]"

**Email:**
- Assunto (com variáveis)
- Corpo do email (editor Tiptap com variáveis)
- Preview renderizado

**Reunião:**
- Tipo: presencial ou online
- Duração sugerida
- Mensagem de convite

#### AC-4: Configuração de ação ao esgotar cadência
**Given** o usuário configurou todos os steps de uma etapa
**When** ele define "Ao esgotar cadência"
**Then** pode escolher:
- **Manter na etapa** (default) — contato fica parado, destacado como "cadência esgotada"
- **Mover para etapa** — selecionar etapa de destino (ex: "Sem resposta", "Arquivo")
- **Notificar responsável** — enviar notificação para o membro responsável pelo lead
**And** a configuração é salva por etapa

### Bloco B — Motor de Automação (Cadence Engine)

#### AC-5: Scheduler de processamento de cadência
**Given** steps de cadência estão configurados e ativos
**When** o scheduler executa (a cada 15 minutos)
**Then** busca contatos elegíveis:
- `currentCadenceStepId` preenchido
- `lastCadenceActionAt` + `daysAfterPrevious` do próximo step <= agora
- Step está ativo (`isActive: true`)
- Contato tem dados necessários para o canal (email para email, phone para ligação, etc.)
**And** para cada contato elegível, dispara a ação do step

#### AC-6: Executor por canal
**Given** o engine identificou um contato elegível para um step
**When** a ação é executada
**Then** o canal correto é usado:

| Canal | Executor | API |
|-------|----------|-----|
| **Ligação** | VAPI voice agent | `POST /api/call-routing/call-contact` com speech do step |
| **WhatsApp** | Twilio | `sendWhatsAppMessage()` com template do step |
| **Email** | Nodemailer | `sendEmail()` com subject + body do step |
| **Reunião** | Notificação | Notificar responsável para agendar manualmente |

**And** variáveis (`{{nome}}`, `{{empresa}}`, etc.) são substituídas com dados do contato
**And** respeita rate limits: máximo 20 ações por lote, 5s entre lotes

#### AC-7: Progressão automática na cadência
**Given** uma ação foi executada com sucesso
**When** o sistema atualiza o contato
**Then**:
- `lastCadenceActionAt` = timestamp da execução
- `currentCadenceStepId` = ID do step executado
- Próximo step é calculado: próximo step do fluxo linear com `order` maior
- Se não há próximo step → cadência esgotada, executar ação de AC-4
**And** um log é criado em `clients/{clientId}/logs`:
- `action`: "cadence_auto_[canal]" (ex: `cadence_auto_whatsapp`)
- `message`: "Cadência automática: [nome do step] via [canal]"
- `metadata`: `{ stepId, stepName, channel, stageId, stageName, templatePreview }`
- `author`: "Sistema (Cadência automática)"

#### AC-8: Tratamento de falhas
**Given** uma ação automática falha (email bounce, ligação não completada, WhatsApp erro)
**When** o erro ocorre
**Then**:
- O contato **NÃO avança** para o próximo step
- Um log de erro é registrado no contato: action="cadence_auto_error"
- Retry automático após 1 hora (máximo 2 retries)
- Após 2 falhas: marcar step como "falhou", avançar para próximo step
- Notificar responsável do contato sobre a falha

### Bloco C — Detecção de Resposta e Avanço de Etapa

#### AC-9: Resposta detectada via webhook de ligação
**Given** o VAPI agent completou uma ligação com resultado positivo
**When** o webhook `/api/vapi/webhook` recebe o resultado
**Then** se o call outcome indica interesse (ex: `REUNIAO_AGENDADA`, `ENVIAR_EMAIL`):
- `lastCadenceStepResponded` = true
- **Pausar cadência** do contato (não executar mais steps nesta etapa)
- Notificar responsável: "Contato [nome] respondeu positivamente na ligação"
**And** o avanço de etapa é **manual** (responsável decide para onde mover)

#### AC-10: Resposta manual registrada
**Given** um membro registra que o contato respondeu (via botão "Marcar resposta" no funil)
**When** a resposta é registrada
**Then**:
- `lastCadenceStepResponded` = true
- Cadência é **pausada** automaticamente
- Log registrado: "Resposta registrada — cadência pausada"

#### AC-11: Reinício de cadência ao mudar de etapa
**Given** um contato é movido para uma nova etapa do funil (manual ou automático)
**When** a nova etapa tem steps de cadência configurados
**Then**:
- `currentCadenceStepId` = ID do primeiro step da nova etapa
- `lastCadenceActionAt` = timestamp da mudança de etapa
- `lastCadenceStepResponded` = false
- Cadência da nova etapa inicia (primeiro step executa após `daysAfterPrevious` dias)
**And** log: "Cadência iniciada para etapa [nome]"

### Bloco D — Dashboard de Execução

#### AC-12: Painel de status da automação em `/cadencia`
**Given** o usuário está em `/cadencia`
**When** ele visualiza a aba "Execução" (nova)
**Then** vê:
- **Contatos em cadência ativa**: quantidade total por etapa
- **Próximas ações agendadas**: lista dos próximos 20 steps a executar (com data/hora prevista)
- **Ações executadas hoje**: count por canal (X ligações, Y WhatsApp, Z emails)
- **Falhas recentes**: lista de ações que falharam (últimas 24h)
- **Cadências esgotadas**: contatos que completaram todos os steps sem resposta

#### AC-13: Controles de pausa/retomada globais
**Given** o usuário está no dashboard de execução
**When** ele gerencia a automação
**Then** pode:
- **Pausar toda a automação** (botão com confirmação) — scheduler para de executar
- **Retomar automação** — scheduler volta a funcionar
- **Pausar cadência de uma etapa específica** — steps daquela etapa não executam
- **Status visual**: badge "Automação ativa" (verde) ou "Automação pausada" (vermelho) no topo

### Bloco E — Integração com Agentes

#### AC-14: Speech do VAPI conectado ao step de cadência
**Given** um step de ligação é executado pelo engine
**When** a chamada é iniciada via VAPI
**Then** o speech/roteiro configurado no step é enviado como `systemPrompt` do agente
**And** o `firstMessage` é montado com saudação + nome do contato
**And** o `metadata` da chamada inclui: `{ cadenceStepId, stageId, contactId }`
**And** o resultado da chamada é processado pelo webhook e atualiza a cadência (AC-9)

#### AC-15: Template de WhatsApp conectado ao step
**Given** um step de WhatsApp é executado pelo engine
**When** a mensagem é enviada via Twilio
**Then** o template configurado no step é usado com variáveis substituídas
**And** se `twilioTemplateSid` está configurado, usa o template aprovado do Twilio
**And** se não, envia mensagem livre (sujeito a limitações do WhatsApp Business)

#### AC-16: Email conectado ao step
**Given** um step de email é executado pelo engine
**When** o email é enviado via Nodemailer
**Then** o assunto e corpo configurados no step são usados
**And** variáveis substituídas com dados do contato
**And** email enviado via `sendEmail()` de `src/lib/email.ts`
**And** registrado em `emailLogs`

### Estratégia de Scheduler
Mesmo padrão de Campanhas (Story 4.1):

**API Route + Cron externo:**
```
cron job (cron-job.org ou Vercel Cron) → POST /api/cadence/process
  → A cada 15 minutos
  → Query contatos elegíveis
  → Executar steps
```

**Vercel Cron Config (vercel.json):**
```json
{
  "crons": [
    { "path": "/api/cadence/process", "schedule": "*/15 * * * *" }
  ]
}
```

### Segurança da Automação — CRÍTICO
Automação descontrolada pode causar:
- **Spam**: enviar centenas de WhatsApp/emails indesejados
- **Custos**: VAPI cobra por minuto de ligação
- **Reputação**: conta Gmail/Twilio bloqueada por abuso

**Safety guards implementados:**
1. `maxActionsPerDay` (default: 100) — hard limit
2. `workHoursStart/End` — não executar fora do horário
3. `enabled` master switch — desliga tudo instantaneamente
4. Rate limiting por lote (20 ações, 5s intervalo)
5. Log completo de tudo para auditoria

### Fluxo Linear Simplificado
O usuário foi claro: **sem branching condicional**. O fluxo é linear:
```
Step 1 → (sem retorno) → Step 2 → (sem retorno) → Step 3 → ... → Esgotou
```
Se respondeu → sai da cadência (pausa). Avanço de etapa é manual pelo responsável.

Isso simplifica **enormemente** a lógica do engine — não precisa de árvore de decisão, apenas sequência ordenada por `order`.

### Testing Standards
- Testar scheduler com contatos em diferentes timings (elegível, não elegível, fora do horário)
- Testar cada executor com mock (VAPI, Twilio, Nodemailer)
- Testar retry após falha (1h depois, max 2x)
- Testar cadência esgotada (ação de mover/manter/notificar)
- Testar pausa por resposta (via webhook e via manual)
- Testar reinício ao mudar de etapa
- Testar safety: maxActionsPerDay atingido
- Testar substituição de variáveis com campos vazios
- Testar feature gate: plano não-Pro vê PlanGate
