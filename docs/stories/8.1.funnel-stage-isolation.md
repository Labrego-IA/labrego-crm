## Story

**Como** gestor de vendas que usa múltiplos funis (ex: Prospecção, Pós-venda, Parcerias),
**Quero** que cada funil tenha suas próprias etapas independentes e que um contato pertença a apenas um funil por vez,
**Para que** eu não veja etapas de um funil aparecendo em outro, nem contatos "vazando" entre funis, garantindo isolamento total de dados.

---

## Contexto

### Estado Atual (BUG)

- **Funis** são criados em `organizations/{orgId}/funnels/{funnelId}` com nome, cor, `isDefault`
- **Etapas** são criadas em `funnelStages/{stageId}` (top-level collection) com `orgId` — MAS o campo `funnelId` **NÃO é salvo** na criação
- **Contatos** têm `funnelStage` (ID da etapa) — MAS **NÃO** têm campo `funnelId`
- **Filtragem** é feita client-side com fallback: `s.funnelId === funnelId || !s.funnelId` — isto faz etapas sem `funnelId` aparecerem em TODOS os funis
- **Resultado**: etapas criadas no Funil A aparecem no Funil B, e contatos do Funil A aparecem no Funil B

### Causa Raiz

1. `handleAddStage()` em `funil/[funnelId]/page.tsx` NÃO inclui `funnelId` no `stageData`
2. Documentos de contato não armazenam `funnelId`, apenas `funnelStage` (ID da etapa)
3. Queries carregam TODAS as etapas/contatos da org e filtram no JS com fallback permissivo

### O que Muda

1. **Criação de etapas** sempre salva `funnelId` obrigatoriamente
2. **Contatos** ganham campo `funnelId` populado automaticamente
3. **Queries Firestore** filtram por `funnelId` no server (não no client)
4. **Migração** corrige dados existentes sem `funnelId`
5. **Enforcement**: contato não pode estar em 2 funis ao mesmo tempo

## Acceptance Criteria

### Bloco A — Modelo de Dados

#### AC-1: Etapas criadas com funnelId obrigatório
**Given** o usuário cria uma nova etapa em qualquer funil
**When** a etapa é salva no Firestore
**Then** o documento da etapa contém `funnelId` igual ao funil onde foi criada
**And** o campo `funnelId` é obrigatório (não `undefined`, não `null`, não string vazia)

#### AC-2: Contatos com funnelId
**Given** um contato é atribuído a uma etapa de um funil
**When** o campo `funnelStage` é atualizado
**Then** o campo `funnelId` é atualizado junto, com o ID do funil correspondente
**And** ambos os campos são sempre atualizados juntos (atomicamente)

#### AC-3: Contato em apenas 1 funil por vez
**Given** um contato já pertence ao Funil A
**When** ele é movido para uma etapa do Funil B
**Then** o `funnelId` muda para Funil B
**And** o `funnelStage` muda para a etapa do Funil B
**And** o contato NÃO aparece mais no Funil A

### Bloco B — Isolamento de Queries

#### AC-4: Etapas filtradas por funil no Firestore
**Given** o usuário acessa `/funil/{funnelId}`
**When** as etapas são carregadas
**Then** a query Firestore inclui `.where('funnelId', '==', funnelId)`
**And** NÃO há fallback client-side do tipo `!s.funnelId`
**And** apenas etapas daquele funil específico são exibidas

#### AC-5: Contatos filtrados por funil
**Given** o usuário acessa `/funil/{funnelId}`
**When** os contatos são carregados no board/kanban
**Then** apenas contatos com `funnelId === funnelId` são exibidos
**And** contatos de outros funis NÃO aparecem

#### AC-6: Contagem de contatos por funil na listagem
**Given** o usuário acessa `/funil` (listagem de funis)
**When** a contagem de contatos por funil é calculada
**Then** conta apenas contatos com `funnelId` igual ao funil
**And** NÃO usa fallback `!s.funnelId` para o funil default

### Bloco C — Migração de Dados

#### AC-7: Migração de etapas sem funnelId
**Given** existem etapas no Firestore sem campo `funnelId`
**When** o script de migração é executado
**Then** etapas sem `funnelId` são atribuídas ao funil default da organização
**And** o script loga quantas etapas foram migradas

#### AC-8: Migração de contatos sem funnelId
**Given** existem contatos com `funnelStage` mas sem `funnelId`
**When** o script de migração é executado
**Then** o script busca a etapa do contato → copia o `funnelId` da etapa para o contato
**And** contatos sem `funnelStage` ficam com `funnelId = ''` (sem funil)
**And** o script loga quantos contatos foram migrados

### Bloco D — Páginas Dependentes

#### AC-9: Conversão filtra por funil corretamente
**Given** o usuário acessa `/conversao` e seleciona um funil
**When** as etapas são filtradas
**Then** apenas etapas com `funnelId` correspondente são usadas
**And** NÃO retorna vazio se todas as etapas têm `funnelId` correto

#### AC-10: Analytics respeita funil selecionado
**Given** o usuário seleciona um funil em `/analytics`
**When** os dados são calculados
**Then** os clientes filtrados respeitam o `funnelId` selecionado

#### AC-11: Cadência respeita funil selecionado
**Given** o usuário seleciona um funil em `/cadencia`
**When** as etapas são carregadas
**Then** apenas etapas do funil selecionado aparecem

#### AC-12: Admin/Funis agrupa corretamente
**Given** o admin acessa `/admin/funis`
**When** as etapas são agrupadas por funil
**Then** cada etapa aparece em exatamente 1 funil (o seu `funnelId`)
**And** NÃO há fallback para o funil default

## Dev Notes

### Padrão de Query Antes vs Depois

**ANTES (bugado):**
```typescript
// Carrega TODAS as stages da org
const unsub = onSnapshot(
  query(collection(db, 'funnelStages'), where('orgId', '==', orgId)),
  (snap) => {
    const all = snap.docs.map(d => ({ id: d.id, ...d.data() }))
    // Filtra no JS com fallback permissivo
    const filtered = all.filter(s => s.funnelId === funnelId || !s.funnelId)
  }
)
```

**DEPOIS (correto):**
```typescript
// Carrega apenas stages do funil específico
const unsub = onSnapshot(
  query(
    collection(db, 'funnelStages'),
    where('orgId', '==', orgId),
    where('funnelId', '==', funnelId),
    orderBy('order', 'asc')
  ),
  (snap) => {
    const stages = snap.docs.map(d => ({ id: d.id, ...d.data() }))
    // Sem fallback — dados já corretos
  }
)
```

### Padrão de Atualização de Contato

Sempre que `funnelStage` mudar, `funnelId` muda junto:
```typescript
await updateDoc(doc(db, 'organizations', orgId, 'clients', clientId), {
  funnelStage: newStageId,
  funnelId: stage.funnelId, // NOVO — sempre junto
  funnelStageUpdatedAt: new Date().toISOString(),
})
```

### Migração — Ordem de Execução

1. **Primeiro**: migrar etapas (setar funnelId do funil default)
2. **Depois**: migrar contatos (derivar funnelId da etapa)
3. **Por fim**: remover fallbacks do código

### Índice Firestore

Pode ser necessário criar índice composto para:
```
Collection: funnelStages
Fields: orgId (ASC), funnelId (ASC), order (ASC)
```
